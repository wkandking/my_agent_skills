# Insight: 领域认知沉淀在项目，方法论沉淀在角色

## 规律

角色的 insights/skills/experience 目录应只包含**方法论和操作性知识**——怎么做研究、怎么组织文献、怎么分解课题。

**领域认知**（某个研究课题的具体发现、理论框架、核心论证）应沉淀在独立项目仓库中。

## 为什么要分开

1. **复用边界不同**：方法论跨课题复用，领域认知只在特定课题内有意义
2. **演化节奏不同**：方法论稳定迭代，领域认知随研究进展快速变化
3. **膨胀风险**：如果领域认知堆在角色目录下，一个深入课题就能产生几十 KB 的 insights，淹没方法论知识，降低索引可检索性
4. **协作边界**：领域认知可能需要独立分享或发布，耦合在 agent 框架仓库里不方便

## 判断标准

| 问题 | 放角色 | 放项目 |
|------|--------|--------|
| 换一个研究课题还有用吗？ | 有用 → 角色 | 没用 → 项目 |
| 是"怎么做"还是"发现了什么"？ | 怎么做 → 角色 | 发现了什么 → 项目 |
| 是操作流程还是理论论证？ | 流程 → 角色 | 论证 → 项目 |
