# 性能实验分析包

## 目标

把一次性能实验整理成稳定、证据完整的分析包，而不是只留下平均值或几张图。

## 最少输入

- 输入样本或候选点
- 数据集定义与规模摘要
- 负载或请求定义
- 原始执行结果
- 聚合摘要

## 输出布局

- `inputs/`
  - 输入样本
  - 数据集摘要
  - 负载定义
  - benchmark 配置原文
  - 生成脚本
- `results/raw/`
  - 原始记录
  - 原始摘要
- `results/derived/`
  - 输入规模表
  - 请求目录
  - 单项对比表
  - 聚合对比表
- `README.md`
  - 实验目标
  - 执行方法
  - 指标定义
  - 执行口径说明
  - 目录说明

## 工作流

1. 整理输入样本，并保留每个样本的规模信息。
2. 整理数据集摘要，至少包含名称、节点数、边数和来源或生成方式。
3. 整理负载或请求定义，包含名称、参数和文本内容；不要只保留外部路径字符串。
4. 明确执行口径，例如 `fixed request` 还是 `timed run`，以及 `query_count` 或 `steady_state_seconds`、`repeat`、`warmup`。
5. 运行实验并保存原始记录，不要只保留平均值。
6. 生成把输入规模和延迟连接起来的单项对比表。
7. 生成用于快速对比的聚合摘要。
8. 编写 `README.md`，说明实验目标、执行方法、指标定义和执行口径。

## 核心要求

1. 必须保留原始记录，不能只留下平均值。
2. 输入规模信息必须和延迟结果一起保存，否则后续很难解释性能差异。
3. 最终交付物应当让别人仅凭分析包目录，就能重建实验输入、执行方法和主要结论。
4. 分析包必须让读者仅凭包内文件就能知道“测了什么数据、怎么跑的、结果为什么可比或不可比”。

## 必补元数据

以下元数据缺任一项，都会显著降低离线分析和报告复核能力：

- 数据集标识：
  - 数据集名称
  - 节点数 / 边数
  - 来源或生成方式
- 执行口径：
  - `fixed request` 还是 `timed run`
  - `query_count` 或 `steady_state_seconds`
  - `repeat`
  - `warmup`
- 场景定义：
  - benchmark YAML、请求模板或等价配置原文
  - 不要只保留外部路径字符串

## 不完整信号

出现以下任一情况，说明分析包不完整：

- 结果包里只有 `summary.json`、`csv`、日志，没有 benchmark 定义
- 写报告时需要回外部仓库猜数据集规模
- 读者无法判断结果是 `fixed request` 还是 `timed run`
- 不同统计口径的结果被放到同一张表里，但没有口径说明
